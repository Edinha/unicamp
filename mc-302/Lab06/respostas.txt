3.1) Fazendo com que a classe carta e magia sejam abstratas (abstract), garantimos que essas
classes nunca serão instanciadas.

Tornando o método usar abstrato, podemos forçar a sua implementação em classes concretas
que herdem de Carta (Card). Essa alteração pode gerar um erro de compilação caso haja uma classe onde
o método não está implementado e retira a obrigatoriedade de existir um método vazio em Carta (Card)
apenas para ser sobreescrito nas classes que a herdam.

3.2)

Como os controladores conhecem apenas as interfaces, este se torna um modelo de baixo acoplamento. Ou seja,
é possével mudar implementações facilmente nesse modelo, visto que para criar uma nova implementação apenas
é necessário que a nova classe "implements" a interface que o controlador conhece. Esse seria o caso também
para criar uma nova estratégia de jogada, além da agressiva.

Com relação as classes internas, é vantajoso pois juntas as classes que fazem sentido apenas em um escopo
fechado, tal como os comparadores da jogada agressiva.
